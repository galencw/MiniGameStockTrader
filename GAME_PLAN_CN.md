# MiniGameStockTrader 中文开发规划

## 1. 项目目标

本项目目标是在 10 天内，以每天约 2-3 小时的节奏，依靠 AI 辅助开发完成一个可玩的 HTML5 小游戏资源包。游戏不依赖后端服务，最终产物应为可直接部署或压缩分发的静态资源目录，包含运行所需的全部代码、配置和本地资源。

首版优先级是做出完整可玩的数值闭环，而不是追求复杂美术或大量内容。玩家需要能从开局进入一局完整游戏，经历普通回合、随机事件、商战回合、对手出牌、股价结算和最终胜负判断。

推荐技术栈为 Vite + React + TypeScript。原因是卡牌、面板、历史记录、图表、配置数据和状态机都适合用组件化和强类型方式组织，也方便多个 AI agent 按模块并行开发。

## 2. 核心体验

玩家扮演做多方，通过出牌影响股价、管理流动资金、维持持仓价值，并在突发商战中抵抗做空对手。对手会在特定窗口概率出现，并尝试通过做空行为压低股价、消耗玩家资产。

游戏画面参考方向：

- 中央是股价走势或 K 线图。
- 图表上方或中间有弹幕式市场消息，营造混乱、幽默和舆论压力。
- 底部展示玩家手牌、资金、持仓、回合按钮和当前状态。
- 一侧展示事件历史、出牌记录、市场情绪和对手状态。
- 商战模式出现时，节奏明显加快，并突出对手行动和股价即时反馈。

首版不需要复杂剧情分支，也不需要真实股票数据。所有股票、事件和角色都可以是虚构内容。

## 3. 一局游戏流程

一局游戏代表一个 8 小时的交易日，但并不是真实时间。内部用 48 个 10 分钟小时间片表示。

- 每局最多 48 个小回合。
- 每 6 个小回合组成 1 个大回合，即 1 个游戏内小时。
- 普通模式下，玩家每个大回合行动一次。
- 商战模式下，每个小回合都发生行动和结算，节奏从 1 小时一回合变为 10 分钟一回合。

推荐初始流程：

1. 开局选择或随机生成一条背景事件，决定当天市场基调。
2. 初始化玩家资金、持仓、股价、市场情绪、牌库和手牌。
3. 进入普通回合，玩家摸牌、出牌、结束回合。
4. 回合结算时，根据玩家卡牌、市场情绪和随机事件影响股价。
5. 每个大回合可能触发系统事件，并显示弹幕消息。
6. 在指定商战窗口内，有概率触发对手登场。
7. 商战模式中，玩家和对手轮流行动，每次行动后股价立即结算。
8. 玩家或对手资金归零时提前结束。
9. 若 48 个小回合结束仍未提前分胜负，则根据最终资产是否达标判断胜负。

## 4. 胜负条件

胜利条件：

- 对手资金或净值被消耗到 0。
- 或者完整坚持到 48 个小回合结束，玩家最终净值达到配置目标。

失败条件：

- 玩家资金或净值被消耗到 0。
- 或者完整游戏结束时，玩家最终净值低于配置目标。

推荐首版默认数值：

- 初始股价：100
- 玩家初始总资产：10000
- 玩家初始现金：10000
- 玩家初始持仓价值：0
- 结算目标净值：20000
- 初始手牌上限：5
- 手牌上限最低值：2

持仓价值可视为玩家的核心生命线。玩家亏损越严重，操作空间越小。建议规则为：相对初始资产每亏损 20%，手牌上限减少 1 张，最低不低于 2 张。

## 5. 卡牌系统

卡牌分为三大类：基本牌、技能牌、事件牌。所有卡牌都应通过配置文件定义，避免把具体卡牌效果写死在 UI 组件中。

### 5.1 基本牌

基本牌用于直接影响股价。

买入：

- 消耗玩家流动资金。
- 小幅推动股价上涨。
- 代表玩家加仓或释放利好买盘。

卖出：

- 回复玩家流动资金。
- 小幅推动股价下跌。
- 代表玩家减仓、释放卖压或回收现金。

推荐默认公式：

- 卡牌面值 20% 对应 2% 股价影响。
- 即 `股价影响百分比 = 卡牌面值百分比 * 0.1 * 杠杆倍率`。

这些数值只作为首版可调参数，不应写死在业务逻辑深处。

### 5.2 技能牌

技能牌主要改变出牌前后的局面状态，不一定直接影响股价。

首版建议包含：

- 增加现金。
- 增加持仓。
- 回复亏损或提高短期承压能力。
- 抽额外牌。
- 临时提高本回合基本牌面值。
- 杠杆效果，使下一张或本回合基本牌效果翻倍。

技能牌需要清楚标记作用时机，例如立即生效、本回合有效、下一张牌有效、下次结算有效。

### 5.3 事件牌

事件牌负责制造幽默、荒诞、舆论反转和市场混乱。事件牌通常不消耗资金，但会明显影响下次股价结算或市场情绪。

首版事件牌方向：

- 大幅提高或降低市场情绪。
- 在下一个结算点额外影响股价。
- 生成一组弹幕文本。
- 修改短期波动率。

事件牌的重点是增强游戏记忆点，但不能让数值完全失控。建议通过配置限制单张事件牌的最大影响范围。

## 6. 市场情绪与随机事件

市场情绪是一个持续影响股价的小型全局变量。建议范围为 -3 到 +3。

- 正情绪会在每次结算时轻微推高股价。
- 负情绪会在每次结算时轻微压低股价。
- 情绪不会代替卡牌效果，只作为背景趋势。

每个大回合开始或结束时，系统可以概率触发随机事件。随机事件可以改变市场情绪、改变短期股价偏置，并生成弹幕。

开局背景事件是特殊随机事件，会定义当天的市场基调，例如：

- 情绪偏多，波动较低。
- 情绪偏空，波动较高。
- 情绪中性，但事件触发概率提高。

弹幕用于表现市场噪音。它只影响展示，不应直接作为核心状态来源。真正影响数值的内容必须落到事件配置或卡牌配置中。

## 7. 商战模式

商战模式是首版最重要的节奏变化点。它代表做空对手突然入场，试图压低股价并消耗玩家资金。

触发方式采用“剧本窗口 + 随机概率”：

- 在若干固定大回合开放触发窗口，例如第 2、4、6 小时附近。
- 每个窗口内按配置概率触发商战。
- 每局不一定触发相同对手，也不一定在相同时间触发。
- 若一局始终未触发商战，可以在后期使用保底概率，避免核心玩法缺席。

商战规则：

- 玩家始终是做多方向。
- 对手始终是做空方向。
- 玩家行动后，股价立即结算。
- 对手行动后，股价也立即结算。
- 商战每轮消耗 1 个 10 分钟小回合。
- 商战可以持续固定轮数，也可以持续到一方资金不足或达到配置退出条件。

对手被击败后，玩家更容易操控股价。可通过降低反向压力、提高玩家卡牌影响或结束商战来表现。

## 8. 对手 AI

对手 AI 必须配置化，方便后续增加不同风格的对手。

推荐对手配置包含：

- 对手名称。
- 初始资金。
- 可用牌组。
- 攻击性参数。
- 保守性参数。
- 触发窗口。
- 出牌权重。
- 对玩家状态的响应策略。

首版 AI 不需要复杂机器学习，只需要规则权重即可。

推荐决策逻辑：

- 当玩家持仓价值高时，更倾向打出强做空牌。
- 当市场情绪已经偏空时，更倾向继续追击。
- 当对手资金较低时，更倾向保守或使用恢复类技能。
- 当玩家现金不足时，更倾向使用压价事件牌。

AI 出牌逻辑要独立于 React UI，放在游戏引擎或策略模块中。

## 9. 技术架构

推荐目录方向：

- `src/game/`：核心游戏状态、状态机、结算逻辑。
- `src/config/`：卡牌、事件、对手、初始数值配置。
- `src/components/`：React 展示组件。
- `src/screens/`：开始界面、游戏界面、结算界面。
- `src/tests/` 或标准测试目录：单元测试和模拟测试。
- `public/` 或 `src/assets/`：本地图片、音效和静态资源。

核心原则：

- 游戏引擎尽量使用纯函数。
- UI 只读取状态并派发动作，不直接修改游戏数据。
- 卡牌和事件通过配置驱动。
- 使用固定随机种子支持复现和测试。
- 所有关键数值集中配置，方便平衡调整。

建议核心类型：

- `GameState`：整体游戏状态。
- `ActorState`：玩家或对手状态。
- `CardConfig`：卡牌配置。
- `EventConfig`：事件配置。
- `OpponentConfig`：对手配置。
- `GameAction`：玩家、对手、系统事件等动作。
- `SettlementResult`：结算结果、股价变化、弹幕、日志。

`GameState` 建议包含：

- 当前阶段。
- 当前模式：普通或商战。
- 当前小回合索引。
- 当前大回合索引。
- 当前股价。
- K 线或价格历史。
- 市场情绪。
- 玩家状态。
- 对手状态。
- 手牌、牌库、弃牌堆。
- 当前激活的临时效果。
- 事件历史。
- 出牌历史。
- 随机种子。

## 10. UI 规划

首版 UI 以清晰可玩为核心，不追求复杂动画, 先用白板或资源库里的临时资源。

主要区域：

- 顶部：当前时间、模式、回合、市场情绪、目标净值。
- 中央：股价走势图或简化 K 线图。
- 图表层：弹幕消息。
- 左侧或右侧：事件记录、出牌记录、结算日志。
- 底部：玩家状态、手牌、出牌按钮、结束回合按钮。
- 商战时：显示对手资金、对手状态、对手行动提示。

交互要求：

- 点击手牌查看效果。
- 可用牌和不可用牌要有明显状态区别。
- 出牌后立即更新预期或实际影响。
- 结束回合按钮只在合法状态可用。
- 商战中要明确显示当前行动方。
- 游戏结束时展示胜负、最终净值、最终股价和关键事件摘要。

## 11. AI Agent 分工建议

为了方便多个 AI agent 并行开发，建议先稳定类型和接口，再分工实现。

### Agent 1：游戏引擎

负责：

- TypeScript 核心类型。
- 游戏状态机。
- 回合推进。
- 股价结算。
- 市场情绪结算。
- 胜负判断。
- 固定随机种子。

边界：

- 不负责视觉样式。
- 不编写具体大量卡牌内容。
- 向 UI 暴露稳定的状态和 action 接口。

### Agent 2：卡牌、事件与数值

负责：

- 基本牌配置。
- 技能牌配置。
- 事件牌配置。
- 开局背景事件。
- 随机市场事件。
- 对手配置。
- 第一轮数值平衡。

边界：

- 不修改 UI 结构。
- 不改核心状态机，只通过配置表达内容。

### Agent 3：React UI

负责：

- 主游戏界面。
- 卡牌手牌区。
- 股价图表。
- 弹幕展示。
- 状态面板。
- 开始界面和结算界面。
- 商战模式视觉提示。

边界：

- 不把游戏规则写进组件。
- 只调用引擎接口派发动作。

### Agent 4：测试与打包

负责：

- 单元测试。
- 多局模拟测试。
- 浏览器冒烟测试。
- `npm run build` 验证。
- 静态资源完整性检查。
- 最终 HTML5 资源包打包说明。

边界：

- 不重写核心规则。
- 发现规则问题时通过测试报告反馈给引擎或数值模块。

## 14. 首版默认假设

- 不接入真实股票数据。
- 不做后端。
- 不做账号系统。
- 不做长期养成系统。
- 不做移动端深度适配，只保证桌面端和常见宽屏可玩。
- 不追求最终美术质量，但需要界面清晰、有基本氛围。
- 所有资源都应进入最终 HTML5 资源包。
- 卡牌、事件、对手和数值都必须尽量配置化。
- 首版目标是一局完整可玩，而不是大量关卡或复杂剧情。

1. 玩家操作的k图绿色柱子经常没有显示, 对手每次都使用了8张牌太多了. 对手也不能一直卖空, 他需要时不时买入.
2. 弹幕区域应该是整个K线图的区域.
3. 游戏规则: 是普通回合是游戏时间一小时. 当有对手进场时(此时应该有提示), 一回合变成了10分钟(只是回合数变多了), 如果没有对手那么一局游戏最多8个回合, 如果第4小时对手加入了, 剩余回合数变成了4*8 = 32 个回合. 如果第6个小时对手被打败了, 那么剩余回合数变成了2小时. 
4. 市场事件面板里显示的, 应该不是游戏的事件, 而是随机的会影响市场情绪的事件. 可以把市场情绪的影响当做是一次出牌. 在玩家或对手结束回合结算的时候, 市场情绪的影响在最后也显示一下.
5. UI显示应该是左下角有对手的头像以及信息, 对手的资金(血条)是能看到的, 但不能看到具体数值. 右下角是玩家头像和信息.
